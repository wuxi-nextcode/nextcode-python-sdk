Workflow Service¶

Service class¶

Service object for interfacing with the Workflow Service API.

This class instance is used to communicate with a RESTFul service. The post_job method creates a new job on the server and find_job and get_jobs allow you to inspect running and past workflow jobs.

Note: This service has not been fully integrated into the SDK and access is still quite raw.

post_job(pipeline_name=None, project_name=None, params=None, script=None, revision=None, build_source=None, build_context=None, profile=None, trace=False, details=None, description=None)[source]¶

Run a workflow job

This is a low-level implementation on top of the workflow service’s POST /jobs endpoint.

Parameters

  • pipeline_name (Optional[str]) – Name of the pipeline to run

  • project_name (Optional[str]) – Name of the project to run this job on, defaults to GOR_API_PROJECT in environment

  • params (Optional[Dict]) – Dictionary of parameters to forward to the job

  • script (Optional[str]) – Git repository url to run (only when server is in development mode)

  • revision (Optional[str]) – Git revision or tag (only when server is in development mode)

  • build_source (Optional[str]) – Source type of the nextflow build (builtin, git, url)

  • build_context (Optional[str]) – Context for build_source, depends on the type.

  • profile (Optional[str]) – Nextflow profile name to use.

  • trace (bool) – Instruct the job to run nextflow trace flags for debugging.

  • details (Optional[Dict]) – Dictionary containing the initial values for ‘details’ in the job

  • description (Optional[str]) – Human readable description of the job

run_local(path, project_name=None, params=None, profile=None, trace=False, details=None, description=None)[source]¶

Run a workflow job from a local nextflow folder

This will package the passed-in folder and upload to a specific bucket that is exported from the workflow service on the cluster (see ‘scratch-bucket’ from the svc.status() method) AWS access keys must be set up correctly to upload to this bucket using typical aws/boto environment and credential management.

Parameters

  • path – Path (relative or absolute) to a folder containing a nextflow script

  • project_name (Optional[str]) – Name of the project to run this job on, defaults to GOR_API_PROJECT in environment

  • params (Optional[Dict]) – Dictionary of parameters to forward to the job

  • profile (Optional[str]) – Nextflow profile name to use.

  • trace (bool) – Instruct the job to run nextflow trace flags for debugging.

  • details (Optional[Dict]) – Dictionary containing the initial values for ‘details’ in the job

  • description (Optional[str]) – Human readable description of the job

wait(max_seconds=None, poll_period=0.5)[source]¶

Wait for a running job to complete.

This is similar to the wait method in the Query Service and will wait by default indefinitely for a workflow job to complete and poll the server regularly to update the local status. When the job is completed (or failed) the method will return the workflow job object.

Parameters

  • max_seconds (Optional[int]) – raise an exception if the job runs longer than this

  • poll_period (float) – Number of seconds to wait between polling (max 10 seconds)

Returns

WorkflowJob

Raises
